### 2.5 网络协议与传输架构是怎么一步一步迭代的

#### 第一阶段：纯 TCP + 换行分隔 JSON

- 时间节点：`2026-07-03 23:13`，提交 `5ff4496`
- 关键实现：`QTcpServer / QTcpSocket`
- 协议形态：`newline-delimited JSON`
- 典型消息：
  - `join`
  - `ready`
  - `place_piece`
  - `start_game`
  - `new_game`
  - `room_state`
  - `game_start`
  - `move`
  - `game_over`

这一版协议的特点是：

- 开发速度快，适合先把五子棋局域网跑通
- 业务语义都直接塞进 JSON 里，调试直观
- 但协议解析、粘包拆包、消息分发都堆在 `NetworkManager` 里，可维护性一般

#### 第二阶段：TCP 控制面 + UDP 广播发现

- 时间节点：`2026-07-04 11:10`，提交 `deca691`
- 新增能力：
  - 局域网 `UDP` 广播发现房间
  - 独立在线服务端 `ServerApp`
  - 在线房间大厅与房间列表

这一阶段不是把主战斗链路改成 UDP，而是：

- 房间连接和游戏同步仍然主要基于原先的 TCP/JSON 思路
- 额外用 UDP 去解决“如何发现局域网房间”的问题
- 网络形态从“手输 IP 直连”升级成“自动发现 + 在线大厅”并存

#### 第三阶段：把 JSON 协议从业务代码里抽出来

- 时间节点：`2026-07-05 17:20`，提交 `04d74f2`
- 关键动作：新增 `linejsonprotocol.cpp/.h`

这一阶段的意义不是换协议，而是：

- 承认早期协议已经可用，但 `NetworkManager` 太臃肿
- 把“换行分隔 JSON”的编解码责任抽成独立模块
- 为后面更彻底的网络层替换做准备

#### 第四阶段：ENet 接入，开始从 TCP 转向 UDP 主链路

- 时间节点：`2026-07-07 01:41`，提交 `3c9ac40`
- 关键新增：
  - `src/network/enetutils.*`
  - `src/network/protocolids.h`
  - `src/game/survivornetcodec.*`
  - `src/game/survivorruntime.*`
  - `src/game/survivorsimulation.*`
  - `src/game/survivorworld.h`

这一阶段说明：

- 项目不再满足于“房间页联机”，开始为实时动作游戏 `Survivor` 建专门的联机运行时
- ENet 已经进入代码主干，不再只是设想
- 协议开始分层：控制消息仍可保持 JSON 语义，但实时战斗包开始走专门编解码

#### 第五阶段：网络层和游戏层一起抽象，正式变成 UDP 为主

- 时间节点：`2026-07-07 05:05`，提交 `c54e4d0`
- 关键动作：
  - 重写 `NetworkManager`
  - 大幅重写 `ServerApp`
  - 新增 `src/common/gamecontrollerbase.*`
  - 继续整理 `roomtypes / types / UI shell`

这一步是整个项目网络形态变化最大的节点：

- 主链路从早期 `QTcpSocket + line JSON` 转向 `ENet / UDP`
- 局域网发现继续保留 `UDP` 广播
- 实时 Survivor 联机开始用更适合高频同步的二进制包 / 多 channel 结构
- 房间控制、大厅控制、实时战斗包不再完全混在一层逻辑里

可以把这一步概括成：

- 早期：`能连上`
- 中期：`能发现房间、能进大厅`
- 后期：`能支撑实时战斗，并且网络层开始可继续演进`

- 本项目不是一次性写完的，而是经历了“本地 MVP -> 局域网联机 -> 多游戏大厅 -> 在线服务端 -> 运行时统一重构 -> Survivor 在线同步攻坚”的连续演进。
