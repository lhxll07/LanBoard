# LanBoard 自动化测试与稳定性验证报告

## 1. 验证目的

本报告记录 LanBoard 桌面版本的自动化回归测试结果，重点验证：

- 房间人数、座位、准备、权限和开局规则。
- 应用协调层在连接、开局、认输、结算和导航过程中的回归行为。
- 局域网发现对 IPv4 地址、多网卡端点、房间聚合和过期清理的处理。

测试不包含物理双机、Android 真机、VPN 实际链路和公网 ECS 压力测试，这些场景需要单独进行人工验收。

## 2. 测试环境

| 项目 | 配置 |
|---|---|
| 操作系统 | Windows |
| Qt | 6.10.3 MinGW 64-bit |
| 编译器 | GCC / MinGW 13.1.0 |
| 构建工具 | CMake 3.30.5、Ninja 1.12.1 |
| 构建类型 | Release |
| 测试框架 | Qt Test、CTest |
| 构建目录 | `build-qt-ascii` |

标准验证命令：

```powershell
cmake --preset qt-mingw-desktop
cmake --build --preset qt-mingw-desktop
ctest --preset qt-mingw-desktop
```

稳定性重复验证命令：

```powershell
ctest --test-dir build-qt-ascii `
  --repeat until-fail:20 `
  --output-on-failure
```

## 3. 测试套件

### 3.1 room-manager-regressions

覆盖房间规则层的稳定行为：

- 玩家 ID 范围、重复加入、容量和 ID 分配。
- 五子棋游戏位和黑白棋子编号。
- 房主固定游戏位、旁观切换、游戏位满员和对局中操作限制。
- 房主开局权限及全员准备条件。
- 五子棋、斗地主、飞行棋和 Survivor 的开局人数规则。
- 切换游戏时清除准备状态并重新整理游戏位。
- 游戏位玩家离开后由旁观玩家补位。
- 对局结束后清除进行中状态和准备状态。
- 房间快照和服务端房间消息字段一致性。

### 3.2 app-controller-regressions

覆盖应用协调和局域网 Host/Client 回归行为：

- 座位切换作用于实际请求者。
- 连接进行中不会被误判为断开。
- 主动模式切换不会产生错误的中间导航。
- 房主开始游戏后本机同步进入游戏状态。
- 局域网房间发现、客户端加入和消息交换。
- 五子棋双方认输及远端认输处理。
- 五子棋最后一步先于 `game_over` 到达客户端。
- 飞行棋结算延迟到当前移动处理结束。

### 3.3 network-discovery-regressions

覆盖局域网发现和地址选择：

- RFC1918、loopback、link-local、multicast 和保留 IPv4 分类。
- 同子网物理网卡优先于 VPN 端点。
- 同一 `roomUid` 的多个端点聚合。
- 忽略本机发布的房间。
- 拒绝缺少有效 `roomUid` 的公告。
- 单端点过期和整房间过期。
- 真实 UDP 数据报接收与解析。

## 6. 验证结论与边界

自动化结果表明，当前桌面构建中的房间规则、应用协调回归点和局域网发现核心逻辑在本机重复运行条件下保持稳定。特别是房间规则已经从间接 UI 验证扩展为独立单元测试，可以在不启动客户端或服务端的情况下快速检查。

上述结果不能替代真实网络环境验收。后续论文和最终答辩仍应补充以下人工证据：

- Windows 双实例或双设备发现与加入房间。
- Windows 与 Android 在同一 Wi-Fi 下互联。
- 启用 VPN 或存在多网卡时的房间去重效果。
- ECS 服务端创建、加入和多人房间状态同步。
